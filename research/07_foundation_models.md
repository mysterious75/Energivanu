# 07 - Pre-Trained Foundation Models for Time Series Forecasting

## Overview

Foundation models for time series forecasting are large pre-trained models that can generate forecasts on unseen datasets with minimal or no task-specific training (zero-shot). This document surveys the major open-source and API-based foundation models, evaluates their suitability for the ENERGIVANU project, and provides practical guidance for integration.

**ENERGIVANU Requirements Recap:**
- 34 input features (multivariate)
- 60-step lookback window
- Prediction horizon: 24 steps
- Inference latency target: <100ms
- Hardware: Kaggle T4 GPU (16GB VRAM)
- Framework: PyTorch

---

## 1. Chronos (Amazon Science)

### Overview

Chronos is a family of pretrained time series forecasting models from Amazon Science. It tokenizes time series values into discrete bins and trains using a language model architecture (T5-based). The latest generation, **Chronos-2**, adds multivariate and covariate support through a novel group attention mechanism.

### Model Variants

| Model | Parameters | Key Features |
|-------|-----------|--------------|
| `amazon/chronos-2` | 120M | Multivariate + covariates, group attention |
| `autogluon/chronos-2-synth` | 120M | Synthetic data trained variant |
| `autogluon/chronos-2-small` | 28M | Smaller Chronos-2 variant |
| `amazon/chronos-bolt-tiny` | 9M | Fast patch-based encoder-decoder |
| `amazon/chronos-bolt-mini` | 21M | Patch-based, 250x faster than original |
| `amazon/chronos-bolt-small` | 48M | Patch-based variant |
| `amazon/chronos-bolt-base` | 205M | Largest bolt variant |
| `amazon/chronos-t5-large` | 710M | Original T5-based (univariate only) |

### How It Works

**Chronos (original):** Time series values undergo scaling and quantization to become a sequence of tokens. A T5 language model is trained on these tokens using cross-entropy loss. Probabilistic forecasts are generated by sampling multiple future trajectories given the historical context.

**Chronos-Bolt:** A patch-based variant that chunks historical context into patches of multiple observations fed into an encoder. The decoder uses these representations for direct multi-step forecasting, generating quantile forecasts. Chronos-Bolt is up to **250x faster** and **20x more memory-efficient** than original Chronos.

**Chronos-2:** Introduces a **group attention mechanism** that enables information sharing across multiple time series within a group. Groups may represent sets of related series, variates of a multivariate series, or targets and covariates. Achieves capabilities through training on synthetic datasets that impose diverse multivariate structures on univariate series.

### Multivariate and Covariate Support

- **Chronos (original):** Univariate only
- **Chronos-Bolt:** Univariate only
- **Chronos-2:** Explicitly supports **univariate, multivariate, and covariate-informed forecasting** zero-shot. On tasks involving covariates, it consistently outperforms baselines by a wide margin.

### Installation

```bash
pip install chronos-forecasting
```

### Code Example: Chronos-2 Inference with Covariates

```python
import pandas as pd
from chronos import Chronos2Pipeline

pipeline = Chronos2Pipeline.from_pretrained("amazon/chronos-2", device_map="cuda")

# Load historical target values and past values of covariates
context_df = pd.read_parquet(
    "https://autogluon.s3.amazonaws.com/datasets/timeseries/electricity_price/train.parquet"
)

# Load future values of covariates
test_df = pd.read_parquet(
    "https://autogluon.s3.amazonaws.com/datasets/timeseries/electricity_price/test.parquet"
)
future_df = test_df.drop(columns="target")

# Generate predictions with covariates
pred_df = pipeline.predict_df(
    context_df,
    future_df=future_df,
    prediction_length=24,
    quantile_levels=[0.1, 0.5, 0.9],
    id_column="id",
    timestamp_column="timestamp",
    target="target",
)
```

### Code Example: Chronos-Bolt (Fast Inference)

```python
import torch
import numpy as np
from chronos import ChronosBoltPipeline

pipeline = ChronosBoltPipeline.from_pretrained(
    "amazon/chronos-bolt-small",
    device_map="cuda",
    torch_dtype=torch.float32,
)

context = torch.tensor(np.random.randn(1, 60))  # batch_size=1, context=60

forecast = pipeline.predict(
    context=context,
    prediction_length=24,
)

print(forecast.shape)  # (batch, num_quantiles, prediction_length)
```

### Fine-Tuning

Chronos-2 is designed as an **inference-only** model. No official fine-tuning instructions are provided. The model is intended to be used zero-shot. For production use, Amazon recommends using Chronos-2 through Amazon SageMaker JumpStart.

The original Chronos (T5-based) can be fine-tuned using the Hugging Face Transformers trainer, but this is not officially documented.

### Performance Benchmarks

- Chronos-2 delivers **>90% win rate** against Chronos-Bolt in head-to-head comparisons
- State-of-the-art on **fev-bench**, **GIFT-Eval**, and **Chronos Benchmark II**
- Largest improvements on tasks that include exogenous features
- Case studies in **energy and retail domains** demonstrated practical advantages

### ENERGIVANU Fit Assessment

| Criterion | Assessment |
|-----------|------------|
| 34 features (multivariate) | Chronos-2: YES (group attention) |
| 60-step lookback | YES (configurable context) |
| <100ms inference | Chronos-Bolt: YES (250x faster); Chronos-2: Unclear |
| 16GB VRAM (T4) | 120M model fits easily |
| Fine-tuning | NOT officially supported for Chronos-2 |

## 2. TimesFM (Google)

### Overview

TimesFM (Time Series Foundation Model) is a pretrained decoder-only foundation model from Google Research. Published at **ICML 2024**, it uses a decoder-only transformer architecture trained on a large corpus of real-world and synthetic time series data.

### Model Details

| Version | Parameters | Context Length | Notes |
|---------|-----------|---------------|-------|
| 1.0 / 2.0 | 500M | 2048 | Archived; `pip install timesfm==1.3.0` |
| 2.5 (current) | 200M | 16k | Latest; PyTorch and Flax versions |

### How It Works

- **Decoder-only transformer** architecture
- Eliminates the frequency indicator that was in earlier versions
- Optional **30M quantile head** for continuous quantile forecasting up to 1k horizon
- Supports normalization, flip invariance, and positivity constraints
- Available in both **PyTorch** and **Flax (JAX)** implementations

### Installation

```bash
git clone https://github.com/google-research/timesfm.git
cd timesfm
uv venv
source .venv/bin/activate

# Choose your backend:
uv pip install -e .[torch]    # PyTorch
uv pip install -e .[flax]     # JAX/Flax
uv pip install -e .[xreg]     # With covariate support
```

### Code Example: Forecasting

```python
import torch
import numpy as np
import timesfm

torch.set_float32_matmul_precision("high")

model = timesfm.TimesFM_2p5_200M_torch.from_pretrained(
    "google/timesfm-2.5-200m-pytorch"
)

model.compile(
    timesfm.ForecastConfig(
        max_context=1024,
        max_horizon=256,
        normalize_inputs=True,
        use_continuous_quantile_head=True,
        force_flip_invariance=True,
        infer_is_positive=True,
        fix_quantile_crossing=True,
    )
)

point_forecast, quantile_forecast = model.forecast(
    horizon=12,
    inputs=[
        np.linspace(0, 1, 100),
        np.sin(np.linspace(0, 20, 67)),
    ],
)

print(point_forecast.shape)    # (2, 12)
print(quantile_forecast.shape)  # (2, 12, 10)
```

### Multivariate / Covariate Support

TimesFM 2.5 supports covariates through **XReg** (external regression), added in October 2025. This requires installing the `xreg` extra. The model processes covariates via a regression-based approach rather than native multivariate attention.

### Fine-Tuning

A fine-tuning example using **HuggingFace Transformers + PEFT (LoRA)** is available at `timesfm-forecasting/examples/finetuning/`. This enables parameter-efficient fine-tuning on custom datasets.

### Integration

TimesFM is integrated into:
- **BigQuery ML** - SQL-based forecasting at scale
- **Google Sheets** - forecasting in spreadsheets
- **Vertex Model Garden** - Dockerized endpoint for production

### ENERGIVANU Fit Assessment

| Criterion | Assessment |
|-----------|------------|
| 34 features (multivariate) | Via XReg (external regression) - indirect |
| 60-step lookback | YES (16k context length) |
| <100ms inference | Likely YES for 200M model |
| 16GB VRAM (T4) | 200M model fits easily |
| Fine-tuning | YES (LoRA via HuggingFace PEFT) |

## 3. Moirai (Salesforce)

### Overview

Moirai is a universal time series forecasting transformer within the **Uni2ts** library from Salesforce AI Research. The original paper was accepted to **ICML 2024 as an Oral presentation**. It provides a unified framework for large-scale pre-training, fine-tuning, inference, and evaluation.

### Model Versions

| Version | Sizes | Notes |
|---------|-------|-------|
| Moirai 1.0-R | small, base, large | Original release |
| Moirai 1.1-R | small, base, large | June 2024 update |
| Moirai-MoE 1.0-R | small, base | Sparse Mixture of Experts variant |
| Moirai 2.0-R | small | August 2025 release |

### How It Works

- **Patch-based transformer** architecture
- Configurable patch sizes: `auto, 8, 16, 32, 64, 128`
- Supports any context length and prediction length
- Uses a mixture distribution approach for flexible probabilistic predictions
- Handles multiple data layouts: `wide`, `long`, and `wide_multivariate`

### Installation

```bash
pip install uni2ts
```

Or from source:
```bash
git clone https://github.com/SalesforceAIResearch/uni2ts.git
cd uni2ts
pip install -e '.[notebook]'
```

### Code Example: Zero-Shot Inference

```python
import torch
from uni2ts.model.moirai import MoiraiForecast, MoiraiModule

MODEL = "Salesforce/moirai-1.1-R-small"
PSZ = "auto"        # patch size: "auto", 8, 16, 32, 64, 128
BSZ = 32            # batch size
CTX = 200           # context length
PDT = 20            # prediction length
NSAMPLE = 100       # number of samples

model = MoiraiForecast(
    module=MoiraiModule.from_pretrained(MODEL),
    prediction_length=PDT,
    context_length=CTX,
    patch_size=PSZ,
    num_samples=NSAMPLE,
    target_dim=1,               # 1 for univariate
    feat_dynamic_real_dim=0,    # number of dynamic real features
    past_feat_dynamic_real_dim=0,
)

predictor = model.create_predictor(batch_size=BSZ)
forecasts = predictor.predict(test_data)
```

### Code Example: Multivariate Setup

```python
model = MoiraiForecast(
    module=MoiraiModule.from_pretrained(MODEL),
    prediction_length=PDT,
    context_length=CTX,
    patch_size=PSZ,
    num_samples=NSAMPLE,
    target_dim=34,                  # 34 for multivariate (our case)
    feat_dynamic_real_dim=0,
    past_feat_dynamic_real_dim=0,
)
```

### Fine-Tuning

Moirai supports fine-tuning via the Uni2ts framework:

```bash
# 1. Prepare custom data
python -m uni2ts.data.builder.simple dataset_name path/to/data.csv --dataset_type wide

# 2. Run fine-tuning with Hydra configs
python -m cli.train -cp conf/finetune run_name=my_run model=moirai_small data=my_data
```

Optional normalization and validation splits are supported.

### Multiple Frequencies

Moirai handles different frequencies implicitly through its unified training framework. The `dataset_type` parameter controls data layout, and the model learns frequency-agnostic representations through its patch-based approach.

### ENERGIVANU Fit Assessment

| Criterion | Assessment |
|-----------|------------|
| 34 features (multivariate) | YES (target_dim parameter) |
| 60-step lookback | YES (configurable context_length) |
| <100ms inference | Likely YES for small model |
| 16GB VRAM (T4) | Small/base models fit easily |
| Fine-tuning | YES (built into Uni2ts framework) |

## 4. Lag-Llama

### Overview

Lag-Llama is the first open-source foundation model for time series forecasting based on a Llama (LLM) architecture. Published in October 2023, it adapts the Llama transformer for time series using lag-based representations.

### Model Details

| Attribute | Value |
|-----------|-------|
| Parameters | **2.45M** (very small) |
| Architecture | Llama-based transformer with RoPE |
| Output | Probabilistic (distributions) |
| License | Apache-2.0 |
| Framework | GluonTS |

### How It Works

- Uses **lag features** as input representations for time series
- Employs **RoPE (Rotary Position Embeddings)** with scaling for extended context lengths
- Outputs probability distributions for each predicted timestep
- Supports any frequency and any prediction length
- Default training context length: 32

### Installation

```bash
pip install lag-llama
```

Or from source:
```bash
git clone https://github.com/time-series-foundation-models/lag-llama.git
cd lag-llama
pip install -r requirements.txt
```

### Code Example: Zero-Shot Forecasting

```python
from lag_llama.gluon.estimator import LagLlamaEstimator
from gluonts.dataset.common import ListDataset
from huggingface_hub import hf_hub_download

# Download pretrained checkpoint
ckpt_path = hf_hub_download(
    repo_id="time-series-foundation-models/Lag-Llama",
    filename="lag-llama.ckpt"
)

estimator = LagLlamaEstimator(
    prediction_length=24,
    context_length=60,
    input_dim=1,
    n_layer=8,
    n_embd=64,
    scaler="mean",
    batch_size=64,
    num_batches_per_epoch=100,
    lr=1e-4,
    ckpt_path=ckpt_path,
)

train_ds = ListDataset(
    [{"start": "2020-01-01", "target": your_time_series_data}],
    freq="H"
)

predictor = estimator.train_model(training_data=train_ds, num_workers=4)
forecasts = predictor.predict(test_ds, num_samples=100)
```

### Fine-Tuning

Lag-Llama supports fine-tuning with these recommended hyperparameters:

| Hyperparameter | Suggested Values |
|----------------|-----------------|
| Context length | 32, 64, 128, 256, 512, 1024 |
| Learning rate | 1e-2, 5e-3, 1e-3, 5e-3, 1e-4, 5e-4 |

Best practices:
- Use a validation split for early stopping with **patience of 50 epochs**
- More fine-tuning data generally yields better results
- Benchmark zero-shot performance first, then fine-tune if needed

### ENERGIVANU Fit Assessment

| Criterion | Assessment |
|-----------|------------|
| 34 features (multivariate) | NO (univariate only, input_dim=1) |
| 60-step lookback | YES (configurable context_length) |
| <100ms inference | YES (only 2.45M parameters) |
| 16GB VRAM (T4) | YES (tiny model) |
| Fine-tuning | YES (GluonTS trainer) |

**Verdict:** Lag-Llama is too small and univariate-only. Not suitable for ENERGIVANU's 34-feature multivariate requirement.

## 5. TimeGPT (Nixtla)

### Overview

TimeGPT is a production-ready, generative pretrained transformer for time series from Nixtla. Trained on over 100 billion data points across retail, electricity, finance, and IoT domains. It is **closed-source** and accessed via API.

### How It Works

- **Encoder-decoder transformer** architecture (inspired by "Attention is All You Need")
- Residual connections and layer normalization
- Final linear layer maps to the forecast window
- **Not based on any existing LLM** - independently trained on time series data

### Installation

```bash
pip install nixtla>=0.7.0
```

### Code Example: Forecasting

```python
import pandas as pd
from nixtla import NixtlaClient

nixtla_client = NixtlaClient(api_key='YOUR_API_KEY')

df = pd.read_csv('your_data.csv')
fcst_df = nixtla_client.forecast(
    df,
    h=24,
    level=[80, 90]  # prediction intervals
)
nixtla_client.plot(df, fcst_df, level=[80, 90])
```

### Code Example: With Exogenous Variables

```python
fcst_df = nixtla_client.forecast(
    df,
    h=24,
    X_df=exogenous_future_df,  # future values of exogenous variables
    level=[80, 90]
)
```

### Key Features

- **Zero-shot inference** without prior training on target data
- **Exogenous variables** support (special dates, events, prices)
- **Multiple series** forecasting simultaneously
- **Cross-validation** built-in
- **Prediction intervals** (configurable confidence levels)
- **Irregular timestamps** handled without preprocessing
- **Distributed computing** compatible with Spark, Dask, Ray

### Performance

- Tested on **300K+ unique series** in zero-shot mode
- Average GPU inference speed: **0.6 milliseconds per series**
- Consistently ranks among top performers across various frequencies

### Pricing

TimeGPT offers a free trial. Production pricing is not publicly documented on the GitHub page - users need to sign up at nixtla.io for details. The NixtlaClient may access services powered by technology from Google, Amazon, IBM, Datadog, and NXAI.

### ENERGIVANU Fit Assessment

| Criterion | Assessment |
|-----------|------------|
| 34 features (multivariate) | YES (exogenous variables) |
| 60-step lookback | YES |
| <100ms inference | YES (0.6ms per series claimed) |
| 16GB VRAM (T4) | N/A (cloud API) |
| Fine-tuning | YES (custom loss functions) |
| Offline/Local | NO (requires API access) |

**Verdict:** Fast and capable, but requires internet access and API costs. Not suitable for offline deployment on Kaggle.

## 7. Recommendation for ENERGIVANU

### Primary Recommendation: Chronos-2 (120M)

**Reasons:**
1. **Native multivariate support** via group attention - handles 34 features natively
2. **Covariate support** built-in for exogenous variables (weather, time-of-day, etc.)
3. **Zero-shot performance** is state-of-the-art on energy domain benchmarks
4. **No fine-tuning needed** - designed for inference-only deployment
5. **120M parameters** fits easily in T4 VRAM
6. **Apache-2.0 license** for commercial use

**Concerns:**
- No official fine-tuning support (if zero-shot performance is insufficient)
- Inference latency for multivariate predictions needs benchmarking

### Secondary Recommendation: Moirai 2.0 (small or base)

**Reasons:**
1. **Native multivariate** via target_dim parameter
2. **Fine-tuning support** built into Uni2ts framework
3. **Flexible architecture** with configurable patch sizes
4. **ICML 2024 Oral** - strong academic validation
5. **Apache-2.0 license**

**Concerns:**
- Moirai 2.0 only released in small size so far
- Less energy-domain-specific benchmarking than Chronos

### Alternative: TimesFM 2.5

**Reasons:**
1. **Largest context length** (16k) for long-range dependencies
2. **LoRA fine-tuning** via HuggingFace PEFT
3. **200M parameters** with quantile head
4. **Google backing** and BigQuery integration

**Concerns:**
- Covariate support via XReg is indirect (not native multivariate)
- Less mature multivariate support than Chronos-2

### Not Recommended

| Model | Reason |
|-------|--------|
| **Lag-Llama** | Univariate only, too small (2.45M), insufficient capacity |
| **TimeGPT** | Requires API access, closed-source, not suitable for offline Kaggle |
| **Original Chronos (T5)** | Univariate only, superseded by Chronos-2 |

## 8. Implementation Strategy

### Phase 1: Zero-Shot Evaluation (Week 1)

1. Install Chronos-2 and evaluate on ENERGIVANU validation set
2. Benchmark inference latency for 34-feature, 60-step lookback
3. Measure zero-shot MAE/RMSE against our LSTM baseline

```python
# Quick evaluation script
from chronos import Chronos2Pipeline
import time

pipeline = Chronos2Pipeline.from_pretrained("amazon/chronos-2", device_map="cuda")

# Benchmark latency
start = time.time()
pred_df = pipeline.predict_df(
    context_df,
    future_df=future_df,
    prediction_length=24,
    quantile_levels=[0.1, 0.5, 0.9],
    id_column="id",
    timestamp_column="timestamp",
    target="target",
)
latency = (time.time() - start) * 1000
print(f"Inference latency: {latency:.1f}ms")
```

### Phase 2: Fine-Tuning (If Needed) (Week 2)

If Chronos-2 zero-shot is insufficient:
1. Try Moirai with fine-tuning on ENERGIVANU data
2. Try TimesFM with LoRA fine-tuning
3. Compare fine-tuned foundation models vs. custom LSTM

```python
# Moirai fine-tuning
python -m uni2ts.data.builder.simple energivanu data/train.csv --dataset_type wide_multivariate
python -m cli.train -cp conf/finetune run_name=energivanu_run model=moirai_small data=energivanu
```

### Phase 3: Hybrid Approach (Week 3)

Consider using foundation model predictions as additional features for the LSTM ensemble:

```python
# Ensemble: foundation model + LSTM
foundation_pred = chronos_pipeline.predict(context)  # (24,)
lstm_pred = lstm_model(lookback)                      # (24,)
ensemble_pred = 0.5 * foundation_pred + 0.5 * lstm_pred
```
